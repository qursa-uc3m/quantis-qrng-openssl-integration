# Quantis QRNG OpenSSL provider

This project exposes an ID Quantique Quantis device as an OpenSSL 3.x/4.x
random provider. It supports the Quantis USB-4M and PCIe-240M through either
the Quantis library or `/dev/qrandomN`.

The provider offers only the name `QUANTIS-QRNG`. Version 1.1 removes the
historical `CTR-DRBG` alias because this provider is not a CTR deterministic
random bit generator. When activated globally, that alias could intercept
unrelated applications fetching OpenSSL's real `CTR-DRBG`. Applications that
explicitly fetch `QUANTIS-QRNG` keep the same API and the default fallback
policy remains unchanged. Calls that supplied previously ignored
personalisation or additional input now fail explicitly.

## Security behavior

By default, unavailable Quantis hardware falls back to the operating system's
`getrandom()` source. This preserves the behavior of earlier releases, but it
means successful output does **not** prove that bytes came from the QRNG. Build
with `-DALLOW_OS_FALLBACK=OFF` when applications must fail closed if the Quantis
source is unavailable.

`XOR_RANDOM=ON` (the default) combines successful Quantis output with
`getrandom()`. If the device is unavailable and fallback is allowed, output is
read once from `getrandom()`.

The direct-device backend opens the configured path when the RAND is
instantiated, rejects a symbolic-link device path and non-character devices,
and retains the descriptor until uninstantiation. Reinstantiate after a device
is connected or reconnected. A source failure puts a fail-closed RAND in
`EVP_RAND_STATE_ERROR`. Provider failures are reported on OpenSSL's error stack
rather than written to the host application's standard error stream.

This is a live random source, not a DRBG. Prediction-resistance requests and a
reseed with no supplied data succeed because the next generation reads a live
source. Personalisation strings, additional input, and externally supplied
reseed entropy are unsupported and are rejected instead of being silently
ignored.

## Build

The direct-device build does not require the Quantis SDK:

```console
cmake -S qrng_openssl_provider -B build \
  -DQUANTIS_LIB=NO \
  -DDEVICE_NUMBER=0 \
  -DOPENSSL_ROOT_DIR=/path/to/openssl
cmake --build build
ctest --test-dir build --output-on-failure
```

For the Quantis library backend:

```console
cmake -S qrng_openssl_provider -B build \
  -DQUANTIS_LIB=YES \
  -DQUANTIS_LIB_DIR=/opt/quantis/Libs-Apps \
  -DDEVICE_TYPE=PCIE \
  -DDEVICE_NUMBER=0
```

Options:

- `QUANTIS_LIB=YES|NO`: use the SDK or `/dev/qrandomN` (default `NO`).
- `DEVICE_TYPE=USB|PCIE`: SDK device type (default `USB`).
- `DEVICE_NUMBER=N`: zero-based device number (default `0`).
- `QUANTIS_DEVICE_PATTERN=/dev/qrandom%d`: direct-device path pattern.
- `XOR_RANDOM=ON|OFF`: combine Quantis and OS randomness (default `ON`).
- `ALLOW_OS_FALLBACK=ON|OFF`: allow OS randomness when Quantis fails
  (default `ON` for backward compatibility).
- `MEASURE_RNG=ON|OFF`: count generated bytes in
  `/dev/shm/random_numbers_shm` (default `OFF`). The counter persists across
  processes/runs until that shared-memory object is removed and saturates at
  `SIZE_MAX` rather than wrapping.

The requested OpenSSL installation must provide both headers and `libcrypto`.
When it is outside the normal search paths, set `OPENSSL_ROOT_DIR`; at runtime,
also make its shared libraries discoverable according to that installation's
instructions.

## Install and use

```console
cmake --install build --prefix /opt/quantis/providers
openssl list -providers -verbose \
  -provider-path /opt/quantis/providers/lib \
  -provider libcustom_qrng_provider
```

Applications use the provider through the EVP RAND API by fetching
`QUANTIS-QRNG`. The `openssl rand` command fetches `CTR-DRBG`, so it no longer
selects this provider after removal of the unsafe compatibility alias.

The sample [`config/openssl.cnf`](config/openssl.cnf) shows how to activate the
provider while retaining OpenSSL's default provider.

## Tests

The CTest integration test loads and unloads the provider, fetches
`QUANTIS-QRNG`, confirms that `CTR-DRBG` resolves elsewhere, checks advertised
state/strength/request parameters, tests disconnected-device fallback, and
checks lifecycle, prediction-resistance, reseed, and unsupported-input
behavior. The strict test points at a regular file to verify that a non-device
path is rejected and that the RAND enters its error state. These tests run
without the Quantis SDK or device.

The following versions were tested locally:

- OpenSSL 3.5.5: passed.
- OpenSSL 3.6.3: passed, including `openssl rand`.
- OpenSSL 4.0.1: passed, including installed-provider `openssl rand`.

The `openssl rand` checks co-load this provider and the default provider and
confirm that the CLI continues to use OpenSSL's real `CTR-DRBG`; they do not
claim that the command's bytes came from Quantis. The matrix separately tests
direct `QUANTIS-QRNG` fetches and CLI provider loading. CI builds and tests
OpenSSL 3.5.5, 3.6.3, and 4.0.1.

Test the strict policy separately with:

```console
cmake -S qrng_openssl_provider -B build-strict \
  -DQUANTIS_LIB=NO -DALLOW_OS_FALLBACK=OFF
cmake --build build-strict
ctest --test-dir build-strict --output-on-failure
```

Hardware-in-the-loop testing remains necessary before release: run the same
test/build with the device connected, then verify throughput, disconnect
behavior, and the `QUANTIS_LIB=YES` backend against the installed SDK.

## Supported devices

- Quantis PCIe-240M
- Quantis USB-4M

The provider does not perform software entropy extraction. That is appropriate
for the PCIe-240M's integrated hardware extraction; USB-4M deployments that need
extraction should add and validate it before relying on raw device output.
